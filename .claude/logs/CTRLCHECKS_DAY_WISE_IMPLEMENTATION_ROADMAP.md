This document is written for both technical and non-technical people. It explains what the team does every day, who does it, how to check that it is really done, what was fixed, and what may still be missing.

## Three Deployable Repos — All Must Stay In Scope

This project is ONE git repo with THREE independently deployable services. Every day of implementation and every test session must account for all three. Never let a chat session scope to only one service.

| Service | Directory | Port | Role |
|---|---|---|---|
| Frontend | `ctrl_checks/` | 5173 dev / 8080 prod | React + Vite SPA; auth via AWS Cognito |
| Backend worker | `worker/` | 3001 | Execution, auth, DB, credential vault, WebSocket, OAuth callbacks |
| AI Generator | `services/ai-generator/` | 3002 | AI workflow generation pipeline extracted from worker |

**Cross-repo contract rule:** Any change to AI generation logic in `worker/` must be mirrored or verified in `services/ai-generator/`. Any new node registered in `worker/src/core/registry/unified-node-registry.ts` must also appear in `ctrl_checks/src/components/workflow/nodeTypes.ts` and in `services/ai-generator/src/lib/` catalog if referenced there.

**Test rotation rule:** Contract test days (Days 33, 123+) must rotate coverage across all three repos. Never let 10+ consecutive days touch only `worker/`. Check `services/ai-generator/src/` and `ctrl_checks/src/` for untested branches at least every 5 days.

---

Developer roles:
- Developer 1: Backend, infrastructure, database, services, security, deployment, monitoring.
- Developer 2: Frontend, testing, QA, user flows, API integration checks, documentation evidence.

Daily meaning:
- Dev 1 builds or fixes the technical foundation.
- Dev 2 tests it from the user side, updates frontend behavior when needed, and records proof.
- Verification explains how anyone can check that the day was completed.
- Fixed today explains what problem was actually solved.
- Missing or watch explains what still needs attention.

Important rules:
- Keep `worker/` running until every new service is stable.
- Keep `worker/src/core/registry/unified-node-registry.ts` as the single source of node behavior.
- Do not add node-specific shortcuts outside the registry.
- Use canary rollout for service extraction: 33 percent, then 66 percent, then 100 percent.
- Test in staging before AWS production migration.
- Keep rollback steps ready for every production change.
- **All three repos must appear in every day's scope check.** Before writing a test or feature, confirm whether `worker/`, `services/ai-generator/`, and `ctrl_checks/` each need a change. If only one needs it, say so explicitly — do not silently skip the others.
- **`services/ai-generator/` is NOT optional.** It owns the AI generation pipeline in production. Its tests, types, and contracts are as important as `worker/` tests.
- **Test commands per repo:**
  - `worker/`: `npx jest path/to/file.test.ts --no-coverage`
  - `services/ai-generator/`: `npx jest path/to/file.test.ts --no-coverage` (run from `services/ai-generator/`)
  - `ctrl_checks/`: `npx vitest run src/path/to/file.test.ts`

### Risks and Mitigation Plans

| Risk | Simple Meaning | Mitigation |
|---|---|---|
| TypeScript errors during service extraction | Moving big code files can break many paths. | Assign specific fix days, fix by folder, keep error logs, and do not start canary until builds pass. |
| Registry logic drifting | Different services may disagree about how workflow nodes behave. | Keep one shared registry package and test every service against it. |
| OAuth redirect updates across 57 providers | Many external app consoles must be updated manually. | Track every provider, keep old redirects active, canary only verified providers, and save proof. |
| Aurora migration rollback unclear | A database migration mistake can cause downtime or data loss. | Test backup, restore, rollback, validation queries, owner decision, and thresholds before production. |
| No safe staging environment | Production becomes the testing place. | Build staging before AWS migration and require staging proof for every cutover. |
| Ollama quality is weak | Cheaper AI may generate broken workflows. | Keep it disabled until JSON, node, edge, credential, and dry-run checks pass. |
| GDPR deletion incomplete | User data may remain in one service after deletion. | Use a deletion coordinator with per-service handlers, retries, audit logs, and verification queries. |
| AWS cost grows too fast | Managed services can become expensive at scale. | Add budgets, alarms, right-size services, and separate always-on cost from burst-to-1M cost. |
| WAF or gateway blocks real users | Security filters may block valid workflow data. | Start in observe-only mode, inspect matches, then enable only safe rules. |
| Load tests hurt real users | Heavy testing can create real incidents. | Prefer staging, use low-traffic windows, set abort thresholds, and keep on-call ready. |
